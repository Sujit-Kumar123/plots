Architecture Reference

Six FastAPI microservices behind a single API Gateway, event-driven CQRS for chat and plot workloads via Apache Kafka, JWT authentication, and a unified PostgreSQL instance with per-service schemas.

Python 3.12 FastAPI 0.115 Next.js 16 React 19 TypeScript 5 PostgreSQL 16 + pgvector Redis 7 Apache Kafka Kubernetes SendGrid · Twilio Bun · uv · Docker

Architecture Diagram

Full request path from browser through Next.js to the API Gateway, across the service tier, and down into the event streaming and data layers.

Amber dashed arrows are Kafka consume events — chat-service and plot-service run an embedded projector consumer group that updates their read models asynchronously, independent of the HTTP request cycle.

Service-to-service auth

All internal HTTP calls must carry x-internal-secret. Downstream services validate it via InternalAuthMiddleware and reject anything without it — even if the caller is within the cluster.

User identity propagation

The gateway decodes the JWT once and forwards x-user-id, x-user-role, and x-user-email headers downstream. No downstream service decodes the JWT — they trust the gateway's forwarded headers.

API Gateway :8000

The sole public entry point for all API traffic. Validates JWTs, caches session lookups in Redis, applies rate limiting, writes audit logs directly to the database, and reverse-proxies requests to downstream services.

Chat Service :8001

Unified command + query handler for chat workloads. A single process runs the FastAPI HTTP server, a Kafka producer (command side), and a background Kafka consumer projector (query side). Alembic migrations are applied at startup.

chat-service
services/chat-service/ · :8001
Writes to chat_write.chat_messages, publishes to chat.events. The chat-projector consumer group updates the chat_read denormalised model asynchronously.
aiokafka Alembic SQLAlchemy async

Plot Service :8003

Unified command + query handler for plot/scene workloads. Same architectural pattern as chat-service: single process with HTTP server, Kafka producer, and embedded plot-projector consumer. Uses a single plot_unified schema with both source-of-truth and projection fields co-located.

plot-service
services/plot-service/ · :8003
Writes to plot_unified.plots. Projector updates element_count and version in place. Bulk create accepts up to 500 plots per request (→ 202 Accepted).
aiokafka Alembic JSONB elements

Endpoints

POST /plots — create (→ 201 + version)
POST /plots/bulk — up to 500 (→ 202)
PUT /plots/{id} — update (→ version++)
DEL /plots/{id} — soft delete (→ 204)
GET /plots — list (paginated)
GET /plots/search?q= — ILIKE name search
GET /plots/{id} — full plot + elements
Config limits: max_plot_elements=10,000, default_page_size=20, max_page_size=100.

Notification Service :8012

Stateless delivery service for transactional email (SendGrid) and SMS (Twilio). Internal callers (auth-service, api-gateway) use the /api/email/* endpoint with x-internal-secret. No database, no Celery — delivery is handled in-process with 3-attempt retry (1s / 2s / 4s backoff, 4xx no-retry).

Payment Service :8011

Stateless placeholder service. No database, no Kafka. The /payments/* router is registered; business logic (Stripe integration) is not yet implemented. All requests authenticated via x-internal-secret.

PostgreSQL Schemas

All services share a single PostgreSQL 16 + pgvector instance. Schemas are created once by scripts/init-schemas.sql (auto-run on first container start). Migrations within each schema are owned by the service that uses it.

SchemaOwnerMigration toolKey tables
public auth-service Alembic (db-migrate init container) users · profiles · sessions · roles · permissions · role_permissions · sheets · audit_logs · file_uploads · password_reset_tokens
chat_write chat-service Alembic (startup auto-migrate) chat_sessions · chat_messages — source of truth
chat_read chat-service (projector) Alembic (startup auto-migrate) chat_sessions (+ message_count, last_message_preview) · chat_messages — denormalised read model
plot_unified plot-service Alembic (startup auto-migrate) plots — write + projection fields co-located (element_count, version)

Backfill: monolith sheets → CQRS

-- One-time migration from legacy sheets table to plot_unified
INSERT INTO plot_unified.plots (id, user_id, name, sheet_w, sheet_d, grid_step, elements, created_at, updated_at)
SELECT id, user_id, name, sheet_w, sheet_d, grid_step, elements, created_at, updated_at
FROM   public.sheets
WHERE  is_deleted = false;

Redis

Redis 7 (password-protected in production) is used by two services for two distinct purposes. No Celery — email delivery moved to notification-service.

ConsumerUseKey pattern
api-gatewaySession validation cache (avoids hitting auth-service on every request)session:{token_hash}
api-gatewayPer-IP rate limiting countersrl:{ip}:{endpoint}
auth-serviceActive session store + revocation listsession:{token_hash}
Graceful degradation: if Redis is unavailable, the gateway falls back to a direct HTTP call to auth-service for session validation rather than failing the request.

Audit logging

The API Gateway owns all audit persistence. AuditMiddleware calls log_action() as a fire-and-forget coroutine on every authenticated request. The audit_admin_router serves GET /api/admin/audit directly from the gateway's own DB connection pool (pool_size=5, max_overflow=10). Auth-service retains only /api/auth/internal/validate-session — it no longer writes or reads audit logs.

Deployment

Docker Compose stacks

docker-compose.dev.yml
Local development — hot-reload, full tooling
  • Source volume-mounted for all services
  • DEBUG=true, Swagger enabled
  • All service ports exposed to host
  • pgAdmin :5050, Kafka UI :8090, Redis Commander :8081
  • Optional Redis password (empty = no auth)
docker-compose.yml
Full production stack — immutable images
  • No source volume mounts
  • DEBUG=false, Swagger disabled
  • Internal services use expose-only (no host ports)
  • CPU + memory limits on all services
  • Redis password required
Kubernetes (k8s/)
Namespace: plots-system
  • StatefulSets: zookeeper, kafka, postgres, redis
  • Deployments: 6 application services + frontend
  • HPAs on all deployments
  • kafka-init Job creates topics once
  • ConfigMap + Secrets for env injection
