Plots

Full-stack monorepo for production-ready applications — FastAPI backend, Next.js 16 frontend, and a CQRS microservices layer built for bulk-data chat and plot workloads, deployed on Kubernetes with Kafka event streaming.

Python 3.12 FastAPI 0.115 Next.js 16 React 19 PostgreSQL 16 Redis 7 Apache Kafka Kubernetes Docker

Architecture

The system is split into two deployment tiers: a monolith (single Docker container handling auth, admin, payments, uploads, notifications) and a CQRS microservices layer dedicated to high-throughput chat and plot operations.

Amber dashed arrows represent asynchronous Kafka event consumption — the query-side projection consumers update their read models independently of the HTTP request/response cycle.

Monolith

Single Docker container: Nginx→FastAPI + Next.js via Supervisord. Handles auth, RBAC, admin, notifications, uploads, payments.

CQRS Layer

5 independent services. Commands write to PostgreSQL and publish Kafka events. Queries read from denormalised projections. Deployed separately on Kubernetes.

Authentication

Kafka Topics

All topics use 3× replication in production (single broker in local dev). Events are keyed to preserve ordering within a session or user. Topics are created once via kafka/create-topics.sh.

TopicPartitionsRetentionKeyPublisherConsumer Group
chat.events127 dayssession_idchat-commandchat-query-projector
chat.bulk.commands61 daysession_idchat-commandchat-query-projector
plot.events1230 daysuser_idplot-commandplot-query-projector
plot.bulk.commands61 dayuser_idplot-commandplot-query-projector
Partition keying: Keying by session_id/user_id guarantees all events for a given session or user land on the same partition, preserving event order within each consumer group without global synchronisation.

Create Topics

# Run once inside the Kafka pod
kubectl exec -n plots-system kafka-0 -- bash < kafka/create-topics.sh

# Or locally with Docker Compose
docker compose -f docker-compose.cqrs.yml up kafka-init

Database Schemas

All services share a single PostgreSQL 16 instance, separated by schema. The public schema belongs to the monolith backend; the CQRS services each own a write schema and a read schema.

SchemaOwnerSideTables
publicMonolithWrite + Readusers, profiles, sessions, roles, permissions, role_permissions, sheets, audit_logs, file_uploads, password_reset_tokens
chat_writechat-commandWrite (source of truth)chat_sessions, chat_messages
chat_readchat-queryRead (projection)chat_sessions, chat_messages (+ last_message_preview, denormalised stats)
plot_writeplot-commandWrite (source of truth)plots (+ version field)
plot_readplot-queryRead (projection)plots (+ element_count, version)

Initialise schemas (first run)

-- scripts/init-schemas.sql (auto-runs via docker-entrypoint-initdb.d)
CREATE SCHEMA IF NOT EXISTS chat_write;
CREATE SCHEMA IF NOT EXISTS chat_read;
CREATE SCHEMA IF NOT EXISTS plot_write;
CREATE SCHEMA IF NOT EXISTS plot_read;

Migrate existing Sheets to CQRS

-- One-time backfill from monolith sheets → plot_write
INSERT INTO plot_write.plots
  (id, user_id, name, sheet_w, sheet_d, grid_step, elements, created_at, updated_at)
SELECT id, user_id, name, sheet_w, sheet_d, grid_step, elements, created_at, updated_at
FROM   public.sheets
WHERE  is_deleted = false;

Kubernetes Deployment

All manifests live in k8s/. Namespace: plots-system. Prerequisite: fill in k8s/secrets.yaml with base64-encoded values before applying.

Deploy order

# 1. Namespace + config
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/secrets.yaml

# 2. Infrastructure (Zookeeper → Kafka → Postgres)
kubectl apply -f k8s/infra/zookeeper.yaml
kubectl rollout status statefulset/zookeeper -n plots-system

kubectl apply -f k8s/infra/kafka.yaml
kubectl rollout status statefulset/kafka -n plots-system

kubectl apply -f k8s/infra/postgres.yaml
kubectl rollout status statefulset/postgres -n plots-system

# 3. Create Kafka topics (once)
kubectl exec -n plots-system kafka-0 -- bash < kafka/create-topics.sh

# 4. Application services
kubectl apply -f k8s/services/
kubectl rollout status deployment -n plots-system

# 5. Ingress + autoscaling
kubectl apply -f k8s/ingress.yaml
kubectl apply -f k8s/hpa/
