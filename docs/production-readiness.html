10 items
01
Rotate all exposed credentials
Google OAuth client secret, Azure AD client secret, and OpenAI API key are stored in .env which is in git history. Rotate them in their respective consoles now, then purge .env from git history with git filter-repo --path .env --invert-paths.

02
Enable TLS / HTTPS end-to-end
The K8s ingress has TLS commented out. Nginx serves HTTP only on port 80. Install cert-manager, uncomment the TLS block in k8s/ingress.yaml, and add HSTS header in docker/nginx/nginx.conf.
Critical
03
Fix Kafka consumer — manual offset commit + dead-letter queue
All four query services use enable_auto_commit=True and log projection failures silently. A failed event is committed and lost forever. Switch to manual commit, commit only after successful projection, and publish failures to a *.dlq topic for replay.
Critical
04
Implement a PostgreSQL backup strategy
The 50Gi postgres PVC has zero backup configuration. A node failure or accidental PVC deletion is unrecoverable. Set up pgBackRest WAL archival to S3/GCS, or switch to a managed database (RDS, Cloud SQL, Azure Database for PostgreSQL).
Critical
05
Add Prometheus metrics to all services
No service exposes metrics. There is no visibility into request rates, error rates, Kafka consumer lag, or database pool exhaustion. Add prometheus-fastapi-instrumentator to every service and ServiceMonitor CRDs for scraping.
Major
06
Wrap Kafka producer send in try/except
send_and_wait() in all command services has no error handling. If the Kafka broker is temporarily unreachable, the HTTP request hangs until timeout with no meaningful error response to the client. Wrap in try/except and return a 503 with a Retry-After header.
Major
07
Add missing database pool parameters to all microservices
The CQRS services (chat-service, plot-service) are missing pool_timeout=30 and pool_recycle=1800 on their create_async_engine() calls. Without pool_recycle, connections become stale after server-side timeouts and cause "server has closed the connection" errors.
Major
08
Add PodDisruptionBudgets and pod anti-affinity
No PDBs exist, so a node drain during a K8s upgrade can evict all replicas simultaneously. No anti-affinity rules means multiple replicas of the same deployment can land on the same node. Add minAvailable: 2 PDBs and preferredDuringSchedulingIgnoredDuringExecution anti-affinity to all deployments.
Major
09
Scale Zookeeper to 3 replicas
Zookeeper runs as a single pod. If it restarts, Kafka loses its coordination layer and all 6 microservices stop producing and consuming events. A Zookeeper quorum requires an odd number of nodes ≥ 3.
Major
10
Enable PostgreSQL SSL in connection strings
No service requires SSL in its DATABASE_URL. Credentials and data travel unencrypted between the application pods and the database. Append ?ssl=require to all connection strings and configure the PostgreSQL container to accept only SSL connections.

Full findings by dimension

4/10 Security ▶
CriticalMicroservices trust the gateway — no auth on internal ports
The API Gateway validates JWT, but the downstream services (chat-service, plot-service, auth-service, notification-service) have zero network-level authentication middleware. If a pod in the cluster can reach chat-service:8001 directly, it bypasses all security controls.
Add a lightweight JWT validation middleware to each microservice, or enforce mTLS via a service mesh.
CriticalWeak default SECRET_KEY values
backend/app/config/auth_config.py defaults to "change-me-to-a-random-string". services/api-gateway/app/config.py defaults to "change-me". If either is deployed without override, JWTs can be forged.
Remove the default entirely — use ... (Pydantic required) so startup fails loudly if the env var is missing.
MajorNo Content-Security-Policy header
docker/nginx/nginx.conf sets X-Frame-Options and X-XSS-Protection but has no CSP, leaving the Next.js frontend vulnerable to injected scripts.
Add Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; ... to Nginx.
MajorCORS allows all methods and all headers
backend/app/main.py sets allow_methods=["*"] and allow_headers=["*"]. This permits TRACE requests and any arbitrary header.
Restrict to ["GET","POST","PUT","PATCH","DELETE","OPTIONS"] and enumerate specific headers.
MinorSwagger UI reachable without auth in debug mode
The API Gateway disables Swagger in prod when DEBUG=False, but verify this setting is actually False in the K8s ConfigMap before deploying.
GoodRate limiting: two-tier Redis-based middleware
Global 300 req/min per IP on all /api/* endpoints plus endpoint-specific limits (login 5/min, register 3/min). Gracefully degrades when Redis is unavailable.
GoodJWT stored in httpOnly cookies
Tokens are not accessible via JavaScript — XSS cannot steal them directly.
5/10 Observability ▶
CriticalNo metrics — Prometheus, OpenTelemetry, or otherwise
Zero visibility into request rates, p99 latencies, error rates, Kafka consumer lag, or database pool saturation. You cannot alert on what you cannot measure.
Add prometheus-fastapi-instrumentator to all 6 microservices. Add a /metrics endpoint. Deploy Prometheus + Grafana. Create ServiceMonitor CRDs.
MajorNo distributed tracing across microservices
Each service has a request_id header, but this ID is not propagated through Kafka messages or forwarded in downstream HTTP calls. There is no way to trace a single user action across api-gateway → command → query.
Adopt OpenTelemetry with the opentelemetry-instrumentation-fastapi and opentelemetry-instrumentation-sqlalchemy packages. Export spans to Jaeger or Tempo.
MajorMicroservice health endpoints don't check dependencies
All four CQRS services return {"status": "ok"} unconditionally. A pod can be marked Ready by K8s while its Kafka consumer is stopped or its database connection is broken.
Add a real readiness check: verify DB connectivity with a SELECT 1 and verify the Kafka consumer task is running before returning 200.
GoodStructured JSON logging with structlog
All log output is structured JSON with request_id context injection, rotating file handlers, and a separate error log. Ready for a log aggregation pipeline (Loki, CloudWatch, Datadog).
GoodMonolith has a proper readiness endpoint
/health/ready issues a live SELECT 1 to PostgreSQL and a PING to Redis before returning 200. This is the right pattern.
6/10 Reliability ▶
MajorNo circuit breaker for gateway → downstream calls
The API Gateway uses httpx.AsyncClient with timeouts, but if all replicas of chat-service are slow, the gateway queues requests until its own connection pool exhausts, cascading the failure.
Add pybreaker or implement a simple exponential-backoff retry with a max of 2 retries and a fast-fail circuit state after 5 consecutive failures.
MajorMissing pool_timeout and pool_recycle on CQRS service databases
The api-gateway sets pool_recycle=1800 to prevent stale connections, but the other microservice databases (chat-service, plot-service) do not. After 30 minutes of inactivity the pool returns dead connections and the service throws errors until recycled.
Add pool_timeout=30, pool_recycle=1800 to every create_async_engine() call in the microservices.
MinorNo terminationGracePeriodSeconds or preStop hooks
Pods are killed immediately on SIGTERM. Kafka consumers may be mid-batch when the pod terminates, causing uncommitted offsets to be reprocessed by the next pod (minor duplication risk).
Add terminationGracePeriodSeconds: 60 and a preStop: exec: sleep 15 hook to all consumer deployments to drain in-flight requests.
GoodKafka producer idempotence enabled
enable_idempotence=True is set on all producers. Combined with acks="all", this guarantees exactly-once delivery per partition even on producer retry.
GoodGraceful consumer shutdown in lifespan
Consumer tasks are cancelled and awaited in the FastAPI lifespan context manager. The consumer loop has a try/finally that stops the Kafka consumer cleanly.
7/10 Kubernetes Configuration ▶
MajorNo PodDisruptionBudgets
During node drain, K8s may evict all replicas of a service simultaneously. With 3 replicas and no PDB, a kubectl drain can take all 3 down at once.
Create a PDB for each deployment with minAvailable: 2.
MajorSingle Zookeeper replica
Zookeeper coordinates Kafka broker elections. One pod means any restart halts Kafka operations across all 6 microservices until it recovers.
Scale Zookeeper to 3 replicas. Zookeeper requires an odd quorum number to elect a leader.
MinorNo pod anti-affinity rules
Replicas of the same deployment can schedule on the same node. A node failure eliminates all replicas of a service simultaneously.
Add preferredDuringSchedulingIgnoredDuringExecution anti-affinity keyed on app label to spread replicas across nodes.
MinorNo rolling update strategy specified
Deployments use the default 25% maxSurge / 25% maxUnavailable. For command services handling writes, even one unavailable pod during a deploy is risky during peak traffic.
Set maxUnavailable: 0, maxSurge: 1 on all write-side deployments to ensure zero-downtime rollouts.
GoodResource requests and limits set on all pods
Every deployment and StatefulSet has CPU/memory requests and limits. The K8s scheduler can bin-pack correctly, and rogue processes can't consume unbounded resources.
GoodHPA configured with CPU + memory thresholds
All 6 microservices have HPAs (min 2–3 → max 8–20 replicas) triggered at 70% CPU and 80% memory. Scales out ahead of saturation.
GoodKafka broker replicas: RF=3, minInSync=2
Kafka topic replication factor 3 with min.insync.replicas=2 means the cluster tolerates one broker failure without data loss or producer stall.
5/10 Error Handling ▶
CriticalKafka consumer silently commits failed projection events
With enable_auto_commit=True, a projection that throws an exception has its offset committed by the broker on the next poll cycle. The event is lost — the query model never reflects the command.
Switch to enable_auto_commit=False. Call await consumer.commit() only after a successful projection. On repeated failure, produce to a *.dlq dead-letter topic.
MajorKafka producer send_and_wait() is unguarded
Command services call await _producer.send_and_wait(...) with no try/except. A transient Kafka timeout surfaces as an unhandled exception, returning a 500 with a raw traceback to the client.
Wrap in try/except, return 503 Service Unavailable with a Retry-After: 5 header when Kafka is temporarily unreachable.
GoodConsistent exception hierarchy with HTTP status mapping
backend/app/common/exceptions.py defines NotFoundException, ForbiddenException, ConflictException, etc. — all mapped to correct HTTP status codes via a registered exception handler. API errors are always structured JSON.
GoodDatabase sessions roll back on exception
All async session context managers in all services call await session.rollback() in the except block, preventing partial writes from being committed on error.
5/10 Database ▶
CriticalNo backup strategy — a single volume deletion is unrecoverable
The 50Gi PVC backing PostgreSQL has no snapshot schedule, no WAL archival, and no restore procedure documented. Accidental deletion or storage failure means total data loss.
Deploy pgBackRest with continuous WAL archival to S3/GCS. Schedule daily base backups. Test restore monthly. Or migrate to a managed database (RDS, Cloud SQL).
MajorNo high-availability — single Postgres instance
A pod crash takes the database down for all 6 microservices until Kubernetes reschedules it and the storage re-attaches. StatefulSet reschedule can take 2–5 minutes.
Deploy Patroni for Postgres HA (3 nodes), or move to a managed service with automatic failover. Query services should use a read replica.
MajorDatabase connections unencrypted in transit
No ?ssl=require in any DATABASE_URL. Credentials and query data travel in plaintext between app pods and the database within the cluster.
Append ?ssl=require to all connection strings. Enable ssl = on in postgresql.conf and provision a TLS certificate for the Postgres pod.
GoodConnection pooling properly sized on the monolith
pool_size=20, max_overflow=10, pool_timeout=30, pool_recycle=1800 — correct values preventing both starvation and connection exhaustion.
Goodpgvector extension available for AI features
The pgvector/pgvector:pg16 image is used in all three stacks and in CI — vector similarity search is supported out of the box and tested in the pipeline.
2/10 Secrets Management ▶
CriticalLive production credentials committed to git history
Google OAuth, Azure AD, and OpenAI credentials exist in .env in the repository's git history. Even after deletion from the working tree, they remain in every clone of the repo until history is rewritten.
1. Rotate all credentials now. 2. Run git filter-repo --path .env --invert-paths. 3. Force-push all branches. 4. Notify all collaborators to re-clone.
MajorK8s Secrets stored as base64, not encrypted
k8s/secrets.yaml uses base64 encoding, which is not encryption. Anyone with kubectl get secret access can decode the values instantly. If etcd is not configured with encryption at rest, the secrets are plaintext on disk.
Enable KMS encryption for etcd (cloud providers offer this natively). Consider sealed-secrets or External Secrets Operator pulling from Vault/AWS Secrets Manager.
MinorNo JWT secret rotation mechanism
Rotating SECRET_KEY immediately invalidates all active sessions. There is no dual-key signing window to allow graceful rotation.
Implement a PREVIOUS_SECRET_KEY that the token validator falls back to. This allows a rotation window during which existing sessions remain valid.
Good.env excluded from git via .gitignore
.gitignore correctly excludes .env and .env.*. The issue is a historical commit, not an ongoing one.
GoodK8s deployments use secretKeyRef — secrets not in ConfigMap
Deployment manifests inject secrets via valueFrom.secretKeyRef, not hard-coded in configmap.yaml. The correct K8s pattern is in place — just needs the encryption layer underneath.
