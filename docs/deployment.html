🚀 CI / CD

Deploying Plots

GitHub Actions pipelines, Docker image publishing to GHCR, and Kubernetes rollouts — all from a single git push.

⚡Pipeline Overview

Every push to main flows through two chained workflows: CI runs first, then CD if CI passes.

ℹ CI and CD are separate workflow files. CI runs on every pull request and push. CD runs only on pushes to main and requires all CI jobs to pass first because both workflows are triggered by the same event and GitHub will queue the CD deployment step after CI completes via needs:.

📄GitHub Workflow Files

Two files live in .github/workflows/.

✓
ci.yml always runs
Runs on every pull request and every push to main. Contains three parallel jobs: backend lint+test, frontend lint+type-check, and Docker Compose config validation. Failing CI blocks the CD deployment.
pull_request push → main
🚀
cd.yml main only
Runs only on pushes to main. Builds 6 Docker images in parallel (frontend app, 5 backend microservices), pushes them to GHCR tagged with the short SHA, then deploys to the plots-system K8s namespace and runs smoke tests.
push → main

🔑Required GitHub Secrets

Set these in Settings → Secrets and variables → Actions in your GitHub repository. GITHUB_TOKEN is provided automatically — never add it manually.

Secret name Used in Description Status
GITHUB_TOKEN cd.yml — push to GHCR Auto-injected by GitHub Actions. Grants write access to the repository's container registry. auto
KUBECONFIG_B64 cd.yml — kubectl deploy Base64-encoded kubeconfig file for your target cluster. Generate with base64 -w0 ~/.kube/config. required
SECRET_KEY k8s/secrets.yaml JWT signing secret for the production app. Store in K8s Secret, not directly in GitHub Actions. required
POSTGRES_PASSWORD k8s/secrets.yaml PostgreSQL password for the production database. Managed via K8s Secret. required
SENDGRID_API_KEY k8s/secrets.yaml Only needed if email notifications are enabled in production. optional
AZURE_STORAGE_CONNECTION_STRING k8s/secrets.yaml Only needed if file uploads are enabled in production. optional
⚠ Sensitive values like POSTGRES_PASSWORD and SECRET_KEY should live in Kubernetes Secrets, not directly in GitHub Actions environment variables. The CD workflow applies k8s/secrets.yaml which reads from these K8s secrets at runtime.

🐍CI — Backend ci.yml

Runs on every PR and push. Spins up a real PostgreSQL 16 + pgvector service container for integration tests.

1
Setup uv + Python 3.12
Uses astral-sh/setup-uv@v4 with caching keyed on backend/uv.lock — subsequent runs skip the install step entirely.
2
Ruff — lint + format check
Runs ruff check . (style/error rules) and ruff format --check . (formatting). Fails fast on first violation.
3
Alembic migrations
Runs alembic upgrade head against the PostgreSQL service container to verify all migrations apply cleanly before tests run.
4
Pytest
Runs the full test suite in backend/tests/ with verbose output. The DATABASE_URL, REDIS_URL, and SECRET_KEY env vars are set to safe CI values.

⚛CI — Frontend ci.yml

Runs in parallel with the backend job. Uses Bun for fast installs and lockfile enforcement.

1
Bun install (frozen lockfile)
bun install --frozen-lockfile fails if bun.lockb is out of date — prevents accidental dependency drift. node_modules is cached on the lockfile hash.
2
ESLint
Runs the project's ESLint config via bun run lint. Catches import issues, unused variables, React hooks violations, and Next.js-specific rules.
3
TypeScript type-check
bun run tsc --noEmit — compiles the entire codebase without emitting files. Catches type errors across all src/ without waiting for a full Bun build.

🐳CI — Compose Validate ci.yml

Validates all three compose files parse and resolve cleanly, in parallel with the other CI jobs.

Runs docker compose config --quiet on all three stack files. This catches: YAML syntax errors, undefined environment variable references, missing service dependency names, and port binding conflicts. Fails fast — doesn't start any containers.

.github/workflows/ci.yml — compose-validate job 
docker compose -f docker-compose.yml config --quiet
docker compose -f docker-compose.dev.yml config --quiet
docker compose -f docker-compose.cqrs.yml config --quiet

☸CD — Deploy to Kubernetes cd.yml

Runs after all 6 build jobs complete. Patches each deployment's image tag, runs Alembic migrations, waits for rollouts, then runs smoke tests.

1
Configure kubectl
Decodes the KUBECONFIG_B64 secret and writes it to ~/.kube/config. All subsequent kubectl commands target the production cluster.
2
Patch image tags on all 6 deployments
Runs kubectl set image for each deployment, setting the new short-SHA tag. Kubernetes immediately begins rolling out new pods (rolling update strategy — no downtime).
3
Run Alembic migrations
Creates a one-off Job from a migrate CronJob template and waits for it to complete (120s timeout). Ensures schema migrations run before new pods serve traffic.
4
Wait for all rollouts
kubectl rollout status on each deployment with a 180s timeout. If any deployment fails to roll out, the workflow fails and the failure annotation points to kubectl rollout undo.
5
Smoke tests
Spawns a temporary curlimages/curl pod inside the cluster to hit /health on both the monolith app service and the API Gateway. Pod is cleaned up automatically after the check.

🏭Manual — Monolith Production Stack

Multi-stage build: Next.js compiled to standalone, Gunicorn + uvicorn workers for FastAPI. Build takes 3–5 min on first run.

production stack
make prod-build       # builds multi-stage image and starts
# Windows:  .\dev.ps1 prod-build

# App served at http://localhost — Swagger UI is disabled in prod.
# PostgreSQL and Redis are internal only (no host port bindings).
⚠ Migrations are not run automatically in the production compose stack. Run them manually from inside the container: docker compose exec app sh -c "cd /app/backend && .venv/bin/alembic upgrade head"

⚙Manual — CQRS Microservices Stack

Starts all 6 microservices, Kafka, Zookeeper, kafka-init (topic bootstrap), Kafka UI, and PostgreSQL.

CQRS stack
make cqrs-build       # builds all 6 service images and starts
# Windows:  .\dev.ps1 cqrs-build

# Services:
#   http://localhost:8000   — API Gateway
#   http://localhost:8090   — Kafka UI
#   localhost:9092          — Kafka broker
#   localhost:5432          — PostgreSQL (CQRS schemas)
⚠ Both the monolith dev stack and the CQRS stack bind PostgreSQL on :5432. Run them one at a time, or change POSTGRES_PORT in your .env before starting the second stack.

Startup order (automatic)

1
PostgreSQL + Zookeeper
Start in parallel. PostgreSQL runs scripts/init-schemas.sql on first start to create the CQRS schemas (chat_write, chat_read, plot_unified).
2
Kafka broker
Starts after Zookeeper is healthy. Exposes :9092 on the host.
3
kafka-init (one-off job)
Waits for Kafka healthy, creates 4 topics (chat.events, chat.bulk.commands, plot.events, plot.bulk.commands), then exits with code 0.
4
6 microservices
Each CQRS service runs alembic upgrade head at startup. chat-service and plot-service also start Kafka consumer background tasks for their internal projectors.

☸Kubernetes — Cluster Setup

All K8s manifests live in k8s/. Deploy to any CNCF-conformant cluster.

1
Create namespace and apply config
bash
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/configmap.yaml
2
Create secrets
Fill in base64-encoded values before applying. Generate with echo -n 'value' | base64.
bash
kubectl apply -f k8s/secrets.yaml -n plots-system
3
Deploy infrastructure (order matters)
bash
kubectl apply -f k8s/infra/zookeeper.yaml
kubectl rollout status statefulset/zookeeper -n plots-system

kubectl apply -f k8s/infra/kafka.yaml
kubectl rollout status statefulset/kafka -n plots-system

kubectl apply -f k8s/infra/postgres.yaml
kubectl rollout status statefulset/postgres -n plots-system
4
Create Kafka topics
bash
kubectl exec -n plots-system kafka-0 -- bash < kafka/create-topics.sh
5
Deploy microservices and autoscalers
bash
kubectl apply -f k8s/services/
kubectl rollout status deployment -n plots-system
kubectl apply -f k8s/hpa/
kubectl apply -f k8s/ingress.yaml

⬆Build & Push Images Manually

For one-off builds outside of GitHub Actions.

bash
REGISTRY=ghcr.io/<your-username>/plots
SHA=$(git rev-parse --short HEAD)

# Frontend app
docker build -t $REGISTRY/app:$SHA -f frontend/Dockerfile .
docker push $REGISTRY/app:$SHA

# Backend microservices
for svc in api-gateway auth-service chat-service plot-service notification-service; do
  docker build -t $REGISTRY/$svc:$SHA services/$svc/
  docker push $REGISTRY/$svc:$SHA
done

⏪Rollback

Kubernetes keeps a rolling history of 3 revisions per deployment by default.

rollback all services to previous revision
# Roll back a single deployment
kubectl rollout undo deployment/api-gateway -n plots-system

# Roll back all services at once
for d in app api-gateway auth-service chat-service plot-service notification-service; do
  kubectl rollout undo deployment/$d -n plots-system
done

# Roll back to a specific revision
kubectl rollout undo deployment/app --to-revision=2 -n plots-system

# Check rollout history
kubectl rollout history deployment/app -n plots-system
⚠ Rollback reverts the running pods but does not reverse database migrations. If the new code ran a schema migration before the rollback, the old code may be incompatible with the migrated schema. Always write backward-compatible migrations and use a separate migration rollback step if needed.

🔁Environment Promotion

How code moves from local → PR → production.

🖥
Local development
Use make dev-build (monolith) or make cqrs-build (CQRS). Changes are hot-reloaded via volume mounts — no Docker rebuild required.
📥
Pull request
Opening a PR triggers the CI workflow. All three jobs (backend, frontend, compose-validate) must pass before merge is allowed. The branch is never deployed.
🔀
Merge to main
Triggers both CI and CD. CI re-runs on the merged commit. If CI passes, CD builds all images, pushes to GHCR, and deploys to the Kubernetes production environment. Requires a GitHub environment approval gate if configured.
